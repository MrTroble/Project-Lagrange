Standards

Every pull request needs to meet certain standards.

Note: Most standards are not used yet by the engine itself at the current date. This is going to be changed as soon as possible.

Classes, Structs, Unions

  • Everything should be put in namespaces
  • Do not use "set namespace", use tge::namespace instead
  • Do not use anything but vectors for dynamic lists

Includes, Headers

  • The first statement in a header file should be
    #pragma once

    Note: This might not be possible for your compiler due to the pragma preprocessor not being a standard

  • The second statements are the includes
  • Subfolders should be separated by a slash (/)
  • ../ is allowed and must be used if you use dependencies
  • Each dependency should be in the root folder or in the dependencies folder
  • Each header must be part of a module

Variables, Macros

  • Every normal variable should be camelCase
  • Every macro or constant variable has to be written in CAPS and snake_case
  • Attributes are at the top of the class or struct
  • Attribute order in classes:
    • const public
    • const protected
    • const private
    • public
    • protected
    • private
  • Macros should be the first definition in the header files after the includes
  • Static variables should not be used (except for dependencies)
  • Use TGE_MACRONAME as the macroname (prefix with TGE_)
  • Declare variables with "extends"

Methods

  • Every method should be snake case
  • Methods are below attributes in classes or structs
  • Methods order in classes:
    • const public
    • const protected
    • const private
    • public
    • protected
    • private
  • Exceptions are not to be used in any case
  • Do not force use inline methods, leave them up to the compiler
  • Use a pPrefix for input parameters
  • Do not override methods

Modules

  • Modules are namespaces
  • Each module is a submodule of tge 
    namespace tge {
    namespace nio {
  • Spaces or underscores cannot be used in module names

Classes, Structs, Unions

  • Avoid using classes and objects, use structs and unions instead
  • Every struct or union has the same standards as classes
  • Every class and struct should be written in UpperCamelCase (Pascal case)
  • They can therefore contain methods
  • Avoid inheriting classes or structs
  • Avoid declaring methods in structs, is permitted if absolutely necessary
